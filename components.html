Component Reference

Interactive and visual primitives powered by MitsuCSS classes and runtime.

Use this page as your implementation checklist. If you are building new UI, try to compose from these components before creating custom styles.

Buttons and Badges

Buttons communicate actions and badges communicate state. Pair them when actions need urgency, counts, or release labels.

Implementation tip: Start with base button/badge class, then add tone and size modifiers. Keep one visual “primary” action per action group.

Helper per item: use .mitsu-btn-group for tightly related actions; icon groups like cancel/alert/done should follow risk order from low to high confirmation.

Helper per item: choose one clear primary action, then secondary tones for supporting actions.

Default iInfo New Active Draft Blocked New Product Hot Sale

Helper per item: badges should be short labels; use sub-badge only for urgent secondary state.

Alert

Alerts communicate status feedback. Use info/success for confirmations and warning/danger for actionable risk states.

Implementation tip: Keep alert copy short and action-oriented. Use dismiss only when the message can safely be hidden.

Helper per item: select alert tone by intent (info/success/warning/danger) and keep dismiss only for non-critical notices.

Tabs + Dialog + Form

These runtime components show progressive interaction patterns. Use tabs for grouped content and dialog for focused decision points.

Implementation tip: Keep tab labels short, keep dialog actions explicit, and use labels for every input field.
